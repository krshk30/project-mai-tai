Shared Schwab Flow

Top-to-bottom view of the live backbone shared by the two bot paths you are focusing on today: the 30-second core bot and the TOS bot, both routed to Schwab.

What This Page Covers The shared orchestration path for the two live Schwab-connected bot flows: market data comes in, the strategy engine ranks and retains symbols, bot watchlists are updated, and OMS executes through Schwab.
Today's Focus This document is centered on the Schwab-connected live path only. It is intentionally not trying to document the Alpaca paper branches you are not focusing on today.
1

Market Data Gateway collects the market view

Full-market snapshot batches come in on the scanner cadence, and active-symbol live trades and quotes keep streaming for the names already on bot watchlists.

↓
2

Strategy Engine scans and ranks names

Each cycle updates the scanner surfaces: momentum alerts, momentum confirmed, five pillars, and top gainers. That creates the candidate set for downstream bots.

↓
3

Feed-retention decides whether a symbol stays active

Confirmed names can remain usable even after they temporarily weaken. The retention layer keeps strong names on the live feed longer, blocks weak ones, and eventually drops dead tape.

↓
4

Symbols are assigned to the Schwab runtime lane

The strategy engine updates each bot watchlist and moves live symbols into the Schwab stream lane used by the Schwab-backed runtimes you care about today.

Schwab stream lane Live watchlist orchestration
↓
5

Bot runtimes warm up bars and evaluate signals

Each runtime receives historical warmup data, then builds bars, indicators, and internal state from the live flow for its own interval and logic family. That includes entry checks, hard-stop checks, scale logic, and close logic.

↓
6

Eligible bots emit trade intents

When a bot sees an open, scale, or close condition, it emits a trade intent that includes its strategy code, account mapping, and routing metadata. The intent can represent entry, hard-stop close, profit-taking scale, normal close, or cancel follow-up logic.

↓
7

OMS/Risk validates and routes to Schwab

OMS is the execution owner. It records the intent, applies risk checks, rejects unsupported or blocked requests, prevents duplicate exits, handles cancel intents, submits the order through the Schwab adapter, and persists broker events and fills.

↓
8

Order events flow back into strategy state and the dashboard

Pending-open, pending-close, scale levels, fills, positions, account-position sync, virtual-position attribution, and service health all round-trip back into the platform so the next scanner and bot cycles operate on updated state.

Read this page as the shared live backbone. The next two pages zoom into the 30-second core bot and the TOS bot without squeezing both flows into one dense diagram.

30s Core Bot Flow

Readable vertical flow for the scanner-to-bot-to-Schwab path used by the 30-second core runtime.

Bot Identity Runtime registration: macd_30s. This is the dedicated 30-second core bot path you are focusing on today.
Routing View This page treats the 30-second core bot as Schwab-connected and focuses on the live path only.
1

Scanner confirms or retains a symbol

The strategy engine promotes a name into the confirmed set or keeps it alive through the feed-retention layer long enough for a valid second-leg or resumed move.

↓
2

The 30s runtime receives the symbol on its watchlist

The watchlist update reaches the MACD 30s bot runtime, along with any blocked-entry state, pending order state, and existing position context.

↓
3

The symbol joins the Schwab live data lane

The 30-second core bot moves the symbol into the Schwab stream subscription lane so the runtime can trade against live Schwab-backed state.

Schwab-backed live path 30-second runtime
↓
4

Recent bar history is warmed up or reseeded

The runtime gets historical bars so the 30s indicator stack does not start cold. On restart, persisted recent bars can be reloaded so the bot resumes with usable indicator context.

↓
5

30-second bars and indicators are built continuously

Incoming trades and quotes update the 30s bar builder. On each completed 30-second bar, the runtime calculates indicators and checks entry, hard-stop, scale, and exit conditions.

↓
6

The bot emits a trade intent when conditions pass

The intent carries strategy_code=macd_30s, the configured account name, the symbol, side, quantity, and any extended-hours routing metadata if needed. Hard-stop exits are emitted here as close intents, not as a separately resting OMS-managed stop order.

↓
7

OMS sends the order through Schwab

The intent is handed to OMS, which runs risk checks, handles cancel/duplicate-exit protections, submits through the Schwab adapter, and records the broker-side order lifecycle as the execution source of truth.

Regular-hours requests default to market orders. Extended-hours routing adds limit-order metadata based on bid or ask pricing.

↓
8

Fills and order events feed back into runtime state

Pending-open, pending-close, scale levels, positions, and next-step watchlist handling are updated so the 30s bot stays in sync with the actual broker and OMS state.

That feedback loop is what keeps hard-stop retries, scale protection, and restart recovery aligned with persisted execution state.

The 30-second core bot is the clearest example of the scanner-driven live path: scanner promotion creates the watchlist target, the runtime builds 30-second bars, and OMS owns the Schwab execution truth.

TOS Bot Flow

Readable vertical flow for the ThinkScript-style runtime from scanner confirmation through Schwab execution and shared-account state updates.

Bot Identity Runtime registration: tos. This is a dedicated TOS-style runtime family, not just a label placed on the generic MACD flow.
Important Distinction The TOS path has its own entry mode. This page is intentionally centered on the Schwab-connected live TOS flow you are reviewing today.
1

Scanner confirmation or retention makes a symbol eligible

The strategy engine decides that the symbol should stay visible to the TOS runtime, either because it is currently confirmed or because retention rules still justify keeping it active.

↓
2

The symbol is pushed onto the TOS watchlist

The TOS runtime receives the same shared orchestration inputs as the other bots: watchlist membership, blocked-entry symbols, pending order state, and position context.

↓
3

TOS shifts into the Schwab stream lane

The TOS runtime subscribes active symbols through the Schwab stream path so live quotes, bars, and order state stay aligned with Schwab execution.

Schwab-backed TOS Dedicated TOS logic
↓
4

1-minute bars are warmed up and kept restart-safe

The runtime builds and restores enough recent bar history so it can resume indicator calculation quickly after service startup or intraday restart.

↓
5

The dedicated TOS entry logic evaluates the bar

The TOS path uses its own ThinkScript-style entry mode rather than simply reusing every generic MACD branch. That keeps the runtime behavior aligned with the intended TOS-style rule set.

High-level examples from the repo docs: MACD cross and reclaim-style paths with VWAP-aware filtering.

↓
6

The runtime emits a TOS trade intent

The trade intent includes strategy_code=tos, the configured account mapping, the TOS quantity setting that OMS will use for execution, and close/scale intent details when TOS exits fire.

↓
7

OMS submits through Schwab and records the shared-account event trail

OMS owns the actual broker submission, risk checks, cancel handling, persistence, and attribution trail. That matters especially when TOS is sharing broker infrastructure with other strategies at the account layer.

Like the 30s path, regular-hours requests default to market orders, while extended-hours routing adds limit-order metadata.

↓
8

Order events, fills, and positions return to platform state

The result comes back into the strategy engine, dashboard, and persistence model so TOS continues with accurate pending state, position ownership, future close logic, and restart-safe execution visibility.

TOS is separate because its decision logic is separate. It shares the live backbone with the 30-second core bot, but its entry path and Schwab execution flow deserve their own readable page.
