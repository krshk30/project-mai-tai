### 🔴 2026-06-16 — 04:00 WATCHLIST-STALENESS RACE (diagnosed; GO-LIVE BLOCKER #4; NOT fixed — recording only)

**Symptom:** at the 04:00 ET session boundary the scanner is clean (fresh pick only, e.g. NXTS) but the **bot
watchlist is stale** = yesterday's symbols + today's fresh pick. v2 shows the same set ONLY because it mirrors
Polygon's published watchlist (commit `463e424`).

**ROOT CAUSE (operator-traced, confirmed) — a millisecond ordering race inside `process_snapshot_batch`:**
the broker-ineligible cache refresh `set_broker_blocked_symbols_by_strategy → _resync_bot_watchlists_from_
current_confirmed` (added in commit `0f5d66e5` / PR #92) runs **BEFORE the scanner session-roll within the same
batch**, so it **re-promotes yesterday's symbols into bot lifecycle (~04:00:00.965)** milliseconds before the
scanner reset (~04:00:00.969). The reset only calls **`set_watchlist([])`**, which — with **lifecycle retention**
— does NOT hard-purge the already-promoted symbols. Net: scanner clean, bot watchlist stale.

**CURRENT IMPACT: harmless (not a safety issue, a correctness/stale-state wart).** The Polygon bot is
**credential-less** (all orders REJECTED: missing Webull keys) and **v2 is paper/simulated** — nothing can lose
money on the stale names today.

**WHY IT'S A GO-LIVE BLOCKER (fix BEFORE the v2 credential transition, not now):** once v2 is **live +
credentialed**, a stale-watchlist race at the 04:00 boundary could have v2 **evaluate/enter YESTERDAY's symbols
at the open** before a clean scanner push lands — exactly the failure you don't want with real money. Better
found now in paper than the morning after going live.

**FIX PACKAGE (for the eventual design-first fix, NOT now) — do BOTH parts:**
- **Part A — correct the race mechanism:** (1) move the **scanner session-roll BEFORE** any broker-blocked /
  manual-stop watchlist resync; (2) on session-roll, **hard-purge non-protected bot lifecycle/watchlist symbols,
  PRESERVING open positions / pending exits** (the carve-out is load-bearing); (3) v2 mirrors **only after** the
  published watchlist is clean.
- **Part B — TIMING SEPARATION (complementary, NOT a substitute):** move the **session reset to ~03:55 ET** so it
  fully completes and state settles **BEFORE** trading logic wakes at 04:00. Today the reset AND bot trade-signal
  generation both fire at ~04:00 simultaneously (the Polygon bot was attempting a signal at the exact instant the
  reset ran) — 03:55 removes that concurrency at the boundary, making it far easier to reason about. Defensively
  sound on its own; Part A still fixes the actual mechanism.
- **DUE-DILIGENCE CHECK before committing to 03:55 (read-only, at fix time):** confirm nothing depends on the
  04:00 timing — specifically, does the scanner need **market data only available at 04:00** to build the FRESH
  watchlist? Clearing stale state at 03:55 shouldn't need new data, but verify the **repopulation** side still
  works (fresh-watchlist build may need 04:00 data). If 03:55 repopulates correctly → ship it with Part A.

**Touches shared watchlist/lifecycle state across ALL bots → design-first** with explicit purge-protection rules +
tests. Does NOT affect the ATR forward-test (that's which ATR entries fire, not which symbols are on the watchlist).
