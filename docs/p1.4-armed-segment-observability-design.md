# P1.3 + P1.4 — armed-segment safety + observability (ship TOGETHER)

**DESIGN ONLY. No code until reviewed.** v2 is live-money; the strategy state path is load-bearing.
Written 2026-07-16, **revised same day** after review caught a design flaw (see §4 — a stopped-process
check is a *stop* gate, not a *start* gate). **P1.3 and P1.4 ship in one restart, not sequentially.**

## 0. What changed in the revision (read first)

The first draft proposed "service-inactive ⇒ GREEN (arms died with the process)" as *the* answer, and a
bootstrap of "cold-start while flat." **Both were wrong:**
- The cap reset does **not** fire on state that survives a restart — it fires on state that gets
  **reconstructed after** one (DB-seed replays bars → `_cw_v2_track` re-arms → `entries_this_flip=0`).
- So a stopped v2 reads GREEN **by construction**, and the check is blind to exactly what materializes
  on the next start. It answers *"is a running v2 safe to restart?"* (yes — it would have caught CPHI),
  **not** *"is a stopped v2 safe to start?"*
- "Flat is safe to start" is the CPHI mistake verbatim — v2 was effectively flat when it re-entered CPHI
  ~6s after boot. **Flat is not safe. That is the entire finding.**

⇒ The **start** question needs its own answer, and the fix (P1.3) must ride the same restart that
deploys the observability (P1.4).

## 1. The problem, precisely

Flat-checks read **positions**. v2's CW entry cap is keyed on an **armed segment** — a symbol that
flipped BUY and is *waiting to enter on the break*, holding **no position**. So an armed segment is
**invisible**, and on restart it is **reconstructed** with `entries_this_flip=0` → the reconstructed
segment can re-enter → **this manufactured the CPHI loss (a restart while flat).**

## 2. What an armed segment IS (code-pinned, `strategy_core/schwab_1m_v2*.py`)

Per-symbol, in `strategy._symbol_states[symbol]`, in-memory only:

| field | meaning |
|---|---|
| `cw_armed` | a BUY flip fired; watching for the break (**the arm**) |
| `cw_segment_high` | running max HIGH since the flip (the trigger the break must exceed) |
| `cw_entries_this_flip` | reclaim counter — the **cap** (max 1, or 2 if reclaim on) |
| `cw_v2_emit_claimed` | an intrabar emit is in flight / filled this segment |

**Armed-and-DANGEROUS** (can re-enter) = `cw_armed && entries_this_flip < max && emit-claim free`.
Note the **claim is set on FILL, not emit** (code: *"claim the segment only when a position OPENS"*),
so today's two emit-without-fill entries (TGHL, ATPC) leave their segments **still armed at n=0** — the
dangerous state. Today's stop is exactly that scenario.

## 3. P1.3 — the fix (rides this restart)

**On boot, after DB-seed reconstruction, mark every reconstructed armed segment `entries_this_flip =
max` (already-used).** ⇒ v2 can only enter on flips that occur **after** boot.
- Cost: one legitimate first-entry on a genuinely-fresh pre-restart segment is skipped.
- Guarantee: a restart can **never** double-enter / re-issue the cap. Fail-closed (operator's shape).
- Alternative (rejected for now): persist live CW segment state across restart = a schema change and it
  re-introduces the "durable state that can lie after a crash" problem. Marking-used is smaller + safer.

## 4. The two gates — they are different questions

| gate | question | answer |
|---|---|---|
| **STOP gate** | is a *running* v2 safe to stop / restart? | flat (positions) **AND** the live snapshot shows **zero dangerous armed segments**. This is what would have caught CPHI. |
| **START gate** | is a *stopped* v2 safe to start? | **NOT readable from the stopped process** (GREEN-by-construction). Answered by **P1.3 being deployed** (reconstructed segments marked used ⇒ safe by construction), **verified by the AFTER-start snapshot** showing zero `n=0` segments. Optionally strengthened by a **dry-run reconstruction** (§5). |

**`service-inactive ⇒ GREEN` is valid ONLY for the STOP gate** (nothing running = nothing to stop
unsafely). It is **not** an answer to the START gate and the check must never present it as one.

## 5. P1.4 — observability (two jobs, build BOTH)

1. **Live gate = the snapshot.** Extend the `StrategyBotStatePayload` v2 already publishes to
   `mai_tai:strategy-state-isolated` with a compact `cw_armed_segments` list
   (`[{symbol, segment_high, entries_this_flip, max_entries, emit_claimed, capped}]`) + `published_at`.
   Built read-only over `_symbol_states` on the publish loop (O(watchlist)).
   - **In-memory snapshot, NOT a durable table** — deliberate and correct: F2's `oms_armed_stops` had to
     *survive* a restart; armed segments must **die with the process**. A durable table would **lie**
     after a crash (show arms that no longer exist). ✅ (review-confirmed)
2. **Durable record = transition logs.** `[V2-CW-ARM] sym=… seg_high=…` / `[V2-CW-DISARM] sym=…
   reason=fill|flip|timeout`. The snapshot answers *"what is armed now?"*; it **cannot** answer *"when
   did this arm?"* — the transitions are the durable audit trail. **Two different jobs.**

**Read side** — `armed_segments_check` (stdlib + `redis-cli`, no app import; F3-style):
- Read latest `strategy-state-isolated` for v2.
- **Freshness gate = 2–3× the state-publish cadence; if stale ⇒ UNKNOWN ⇒ fail closed (RED).** Never
  read "no arms" off a stale snapshot (the false-flat lesson).
- non-empty dangerous list ⇒ RED (list them); empty+fresh ⇒ GREEN.
- **service-inactive ⇒ GREEN — but label it "STOP-safe; says nothing about START."**

## 6. ⭐ Acceptance test — P1.4 makes P1.3's bug directly visible (operator)

`armed-and-dangerous = cw_armed && entries_this_flip < max`. Across a restart the cap reset is the
observable flip:

```
BEFORE restart:  TGHL armed, entries=1   → NOT dangerous (capped)
AFTER  restart:  TGHL armed, entries=0   → DANGEROUS      ← this flip IS the cap reset
WITH P1.3:       TGHL armed, entries=max → safe (marked used)  ← P1.3 working, in the act
```

**The test:** snapshot the armed segments before the stop and after the start; a reconstructed segment
that comes back `entries < max` is P1.3 failing. This is P1.3's acceptance test — **catch the cap reset
in the act, or prove it can't happen.** (Bootstrap caveat: for *this* first deploy we have no
before-snapshot — see §7.)

## 7. Bootstrap — the window is NOW, and it is not "flat = safe"

The fleet is **stopped right now** — no chicken-and-egg while nothing runs. **This is the window.** But:
- **P1.3 must ship in the SAME restart that deploys P1.4**, or the deploy restart itself re-issues every
  reconstructed cap (today's TGHL/ATPC segments are armed at n=0 in the bars).
- We have **no before-snapshot** for today (the field doesn't exist yet, and the in-memory state is
  already gone). So the acceptance for *this* deploy is the **AFTER-start snapshot with P1.3 on**: every
  reconstructed armed segment must show `entries=max` (capped), **zero at n=0**. If any come back n=0,
  P1.3 is wrong — stop, do not let it trade.
- **Do NOT rely on "flat" at any point.** The gate for resuming trading is the after-snapshot, not the
  position count.

## 8. ⚠ Register item (surfaced by the watchdog check; NOT tonight's work)

`fleet_health` check #1 is hardcoded `strategy_code='polygon_30s'` (the paper engine); oms-liveness
watches the **OMS** heartbeat. ⇒ **no health check watches v2's or ORB's function at all** — in a
system whose founding principle is *"components report healthy while their function is dead."* The
real-money bots have no functional monitor. **Register as its own item.**

## 9. Tests (each must fail before the change)

1. `cw_armed=True, entries=0` ⇒ appears in `cw_armed_segments` as dangerous.
2. capped (`entries>=max`) ⇒ `capped=true`, not dangerous.
3. `armed_segments_check`: non-empty fresh ⇒ RED; empty fresh ⇒ GREEN; **stale ⇒ RED**;
   service-inactive ⇒ GREEN **labelled STOP-only**.
4. **P1.3:** boot with a reconstructible armed flip in the seed bars ⇒ post-boot `entries==max`, and a
   post-boot break does **not** enter. (The acceptance test, as a unit test.)
5. building the list does not mutate `_symbol_states`.
6. transition logs fire on arm and on each disarm reason (fill/flip/timeout).

## 10. Rollout

- P1.3 + P1.4 land together, one PR, design-review first (v2 live-money discipline). Deploy = the
  attended restart while the fleet is already down (today's window), gated by the **after-start
  snapshot** (§7), not by flatness.
- The read-only `armed_segments_check` can ship to the F3 cron independently (no restart) so a *running*
  v2 sitting on a dangerous arm becomes visible on the dashboard/ntfy, not only at restart time.
- Rollback: P1.3 behind a flag (default off = pre-P1.3 behavior); P1.4 snapshot field is additive.
