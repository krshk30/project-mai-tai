# P1.4 — armed-segment observability (design)

**DESIGN ONLY. No code until reviewed.** v2 is live-money; the strategy state path is load-bearing.
Written 2026-07-16. This is the **unblocker for P1.3** — you cannot safely deploy the cap-reset fix
without first being able to *see* the state it acts on.

## 1. The problem, precisely

The fleet-flat check reads **positions** (`oms_managed_positions` + `virtual_positions`). But v2's CW
entry cap is keyed on an **armed segment** — a symbol that has flipped BUY and is *waiting to enter on
the break*, holding **no position yet**. So:

- An armed segment is **invisible** to every "is it flat / safe to restart?" check.
- P1.3: a restart **reconstructs** armed segments (db-seed → `on_bar` → `_update_atr_state`) and
  **re-issues the per-segment entry cap**, so a reconstructed segment can re-enter → **this
  manufactured the CPHI loss** (a restart *while flat*).
- ⇒ **"flat" ≠ "safe to restart."** Until armed segments are observable, every v2 restart is luck.

**Observability is P1.4; the behavioral fix is P1.3.** P1.4 lets you (a) *decide* a restart is safe and
(b) *verify* P1.3 actually neutralizes reconstructed segments. Do P1.4 first.

## 2. What an armed segment IS (code-pinned, `strategy_core/schwab_1m_v2*.py`)

Per-symbol state in `strategy._symbol_states[symbol]`, in-memory only:

| field | meaning |
|---|---|
| `cw_armed: bool` | a BUY flip fired; watching for the break (**the arm**) |
| `cw_segment_high: float` | running max HIGH since the flip (the trigger the break must exceed) |
| `cw_entries_this_flip: int` | reclaim counter — the **cap** (max 1, or 2 if reclaim on) |
| `cw_v2_emit_claimed: bool` | dedup: an intrabar emit is in flight / filled this segment |
| `cw_v2_bars_since_exit: int` | reclaim-gap counter |

**An armed-and-dangerous segment** (the one that can re-enter after a restart) is:
`cw_armed == True` **AND** `cw_entries_this_flip < _cw_v2_max_entries_per_flip` **AND** the emit-claim
is free. That triple is the observable we must surface.

## 3. Design — surface it through the state snapshot v2 already publishes

v2 already runs `_publish_bot_state()` → `StrategyBotStatePayload` → the
`mai_tai:strategy-state-isolated` Redis stream (for the dashboard), on `_state_publish_loop`. **Extend
that payload; do not add a new table.**

- **Why the snapshot, not a durable table (unlike F2's `oms_armed_stops`):** F2 needed state that
  *survives a restart* (a broker stop that outlives the process). Here we want the **opposite** — armed
  segments **die with the process**, and that is correct (a stopped v2 has no armed segments). So the
  observable must reflect **live in-memory state**, which is exactly what a fresh snapshot is. A durable
  table would *lie* after a crash (show arms that no longer exist).
- **Add to the payload** a compact `cw_armed_segments` list: `[{symbol, segment_high, entries_this_flip,
  max_entries, emit_claimed, armed_at_bar}]`, built by iterating `strategy._symbol_states` for
  `cw_armed and not fully-capped`. Plus a monotonic `snapshot_seq` and `published_at` (event-time) so a
  reader can tell **fresh from stale**.
- **Guard:** building this list must be read-only over the state dict on the publish loop (no mutation,
  no lock contention with the entry path). It is O(watchlist) — tiny.

## 4. The read side — a check that answers "any armed segments?" fail-closed

New read-only check `armed_segments_check` (stdlib + `redis-cli`, no app import — same shape as the F3
checks and `assert_fleet_flat.sh`):

1. Read the latest `strategy-state-isolated` entry for v2 from Redis.
2. **Freshness gate:** if `published_at` is older than N s (e.g. 30s) → **UNKNOWN → fail closed**
   (never report "no arms" off a stale snapshot — that is the false-flat lesson, [[false-flat]]).
3. If `cw_armed_segments` is non-empty → **RED, list them** (symbol + entries/cap).
4. If empty **and** fresh → GREEN "no armed segments."
5. **v2 service inactive** (stopped): the stream has no fresh entry → the check must special-case
   `systemctl is-active schwab-1m-v2 == inactive` ⇒ **GREEN "no arms (process down; segments died with
   it)"** — this is the *only* safe way to read absence, and it is why the stop-script already treats
   "unit inactive" as "no armed segments." Do NOT infer no-arms from a stale snapshot alone.

**Integrate into the restart pre-flight:** the safe-to-restart gate becomes
**`assert_fleet_flat` (0 positions) AND `armed_segments_check` (0 arms or process-down)**. Wire it into
`stop_bots_and_clear_queue.sh` and the F3 fleet-health cron (so a running v2 sitting on an armed segment
is visible on the dashboard/ntfy, not just at restart time).

## 5. Edge cases

| case | expected |
|---|---|
| v2 stopped | `is-active=inactive` ⇒ GREEN (arms died with the process). The stop-script path. |
| snapshot stale (>30s) while v2 active | UNKNOWN → **fail closed** (RED). A live-but-frozen v2 must not read as "no arms." |
| armed but already capped (`entries>=max`) | not surfaced as *dangerous* (can't re-enter) — but include with a `capped=true` flag for completeness. |
| reclaim OFF (current: `max_entries=1`) | a segment that already entered once is capped ⇒ not dangerous. Only `cw_armed && entries==0` is the re-entry risk. |
| ORB | out of scope — ORB's "arm" is the native stop, already durable via `oms_armed_stops` (F2). P1.4 is **v2 CW-only**. |
| multiple armed symbols | list all; the count is what gates the restart. |

## 6. Relationship to P1.3 (the sequence)

- **P1.4 (this) = SEE armed segments.** Ships first. Pure observability — the entry/exit path is
  untouched (only the snapshot payload grows + a read-only check). Lower risk → design-review, then a
  small PR.
- **P1.3 (next) = don't re-issue the cap on reconstructed arms.** Fail-closed shape (operator): *on
  boot, mark every reconstructed segment `entries_this_flip = max` (already-used)* ⇒ v2 can only enter
  on flips that occur **after** boot. Costs one legit first-entry on a pre-restart segment; can **never**
  double-enter. **P1.4 is how you prove P1.3 works**: restart with an armed segment present → the
  check shows it → confirm v2 does *not* re-enter it.

## 7. Tests (each must fail before the change)

1. A symbol with `cw_armed=True, entries=0` appears in the published payload's `cw_armed_segments`.
2. A capped symbol (`entries>=max`) is flagged `capped=true`, not counted as dangerous.
3. `armed_segments_check`: non-empty fresh snapshot ⇒ RED; empty fresh ⇒ GREEN; **stale ⇒ RED
   (fail-closed)**; **service-inactive ⇒ GREEN**.
4. Building the list does not mutate `_symbol_states` (read-only over a copy of the keys).
5. Payload with no CW-v2 enabled (`_cw_v2_enabled=False`) ⇒ empty list, no error.

## 8. Rollout

- Snapshot change is additive/observational (flag-guardable via the existing CW-v2 flag). Deploy needs
  a v2 restart — but **that restart is itself gated by P1.3 not yet existing**, so land P1.4's snapshot
  change **together with, or after, P1.3**, or do the first P1.4 deploy as a deliberate attended
  cold-start while flat (the very operation P1.4 exists to make safe — bootstrap chicken-and-egg noted).
- Read-only check ships independently (no restart; cron picks it up), and works immediately for the
  *service-inactive* case even before the snapshot field exists.

## 9. Open questions for review

1. Snapshot field vs. a dedicated lightweight `mai_tai:v2-armed-segments` key — the snapshot reuses
   plumbing but couples cadence to the dashboard loop; a dedicated key could publish on arm/disarm
   *transitions* (fewer stale-window worries). Lean: reuse the snapshot + a tight freshness gate.
2. Freshness threshold N (30s?) — must be > the state-publish interval, < a meaningful stall.
3. Do we also want a `[V2-CW-ARM]` / `[V2-CW-DISARM]` transition log for post-hoc audit (cheap, and it
   would have made the CPHI post-mortem trivial)? Recommend yes, alongside.
